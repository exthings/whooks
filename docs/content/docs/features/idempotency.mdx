title: Idempotency
description: Idempotency of webhooks

Whooks implements [idempotency](https://en.wikipedia.org/wiki/Idempotence) to allow you to retry requests safely without the risk of duplicating operations. This feature becomes particularly valuable when network disruptions occur and you are unsure if an API request was successfully processed.

<Callout type="info">

This documentation covers making idempotent API requests to Whooks. If you are looking for guidance on how your webhook consumers can prevent processing duplicate messages, see the [receiving idempotency (deduplication)](#receiving-idempotency-deduplication) section at the bottom of this page.

</Callout>

To make an API request idempotent, include a `uid` parameter in the request body. The value of this key must be a unique identifier generated by your client application. Use a high-entropy string generation technique (like UUID v7) to prevent accidental collisions.

<Tabs items={["cURL"]}>
<Tab value="cURL">

```shell
curl "https://whooks.exthings.org/v1/events" \
    -H "Accept: application/json" \
    -H "Content-Type: application/json" \
    -H "Authorization: Bearer {API_KEY}" \
    --data-binary @- << EOF
        {
        "uid": "fk0113ybcxknwrrjz3311w1v",
        "consumer_id": "consumer_01kg14nxkbfxgs1fv576yh871z",
        "project_id": "project_01kf4s8r2wf4crny60y8gbeen8",
        "topic": "transaction.completed",
        "data": {
            "id": "019bb3b7-57f7-7b28-9ad1-ce122c7a66c0",
            "status": "completed",
            "amount": "15"
        },
        "metadata": {
            "transaction_id": "01KESVHY8HACK7HGVYPRP90WS0"
        },
        "tags": ["019bb3b8-198d-7b7d-92af-d6dcb0ee7aad"]
        }
    EOF
```

</Tab>
</Tabs>

Behind the scenes, Whooks handles idempotency by temporarily caching the HTTP status code and response payload of the initial successful request associated with a specific idempotency key. Any subsequent API calls made with the identical authentication token and idempotency key will receive the exact same cached response for up to 1 hour.

Note: Idempotency keys are exclusively supported on `POST` requests.

## Receiving idempotency (deduplication)

Whooks guarantees "at least once" delivery for webhooks. Consequently, transient problems like network timeouts or temporary unreachability might result in the same webhook event being delivered to your endpoint more than once.

For many applications, processing duplicate events is harmless. However, if your system strictly requires "exactly once" processing, your webhook consumers must implement their own deduplication logic.

To facilitate this, Whooks attaches a `webhook-id` header to every outgoing webhook. This identifier remains consistent across all retry attempts for a given message. Endpoint implementations can use this `webhook-id` to filter out duplicates—for instance, by caching the ID in Redis with a 24-hour expiration and verifying that the incoming `webhook-id` hasn't been processed recently.
