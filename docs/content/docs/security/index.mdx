# Webhook Security

Webhook endpoints receive requests from external systems.
Because of that, you should never process a webhook request without verifying that it actually came from Whooks.
Whooks signs webhook requests follow standard webhooks specification so your system can verify their authenticity before processing the event.


## Why webhook security matters

Without verification, your webhook endpoint could accept requests from any source.
This can lead to:

- Unauthorized actions
- Data manipulation
- Fake webhook events
- Incorrect business logic execution
- Security vulnerabilities

<Callout type="warn">
  Signature verification helps ensure that the request was sent by Whooks and that the payload was not modified in transit.
</Callout>


## Signature header

Whooks includes a id, timestamp and signature header with each event delivery attempt.

The header contains:

- `webhook-id`: the unique webhook identifier described in the sections above.
- `webhook-timestamp`: integer unix timestamp (seconds since epoch).
- `webhook-signature`: the signature(s) of this webhook.

Example:

```http
webhook-id: attempt_01kqg68sa2ekgs6c2p9vw9rknw
webhook-timestamp: 1674087231
webhook-signature: v1,K5oZfzN95Z9UVu1EsfQmfVNQhnkZ2pj9o9NDN/H/pI4=
```

<Callout>
The signature is generated using the delivery attempt body and a signing secret.
</Callout>


## Signing secret

The signing secret is a private value shared between Whooks and your system.
Your system uses this secret to calculate the expected signature and compare it with the signature received in the webhook request.

Keep the signing secret secure.

Do not expose it in:
- Frontend code
- Public repositories
- Client-side applications
- Logs
- Error messages


## How verification works

A typical verification flow looks like this:


### Receive webhook request [step]

### Read raw request body [step]

### Verify timestamp [step]

### Calculate expected signature using signing secret [step]

### Compare expected signature with received signature [step]

### Reject request if verification fails [step]

### Process event if verification succeeds [step]

---

## Use the raw request body

When verifying signatures, always use the raw request body.
Do not verify the signature using parsed JSON.
Parsing and re-stringifying JSON can change spacing, ordering, or formatting, which may cause signature verification to fail.

Correct approach:

```txt
Raw request body → verify signature → parse JSON → process event
```

Incorrect approach:

```txt
Parse JSON → stringify again → verify signature
```

## Example verification

Example using Node.js:

```js
import crypto from "crypto";

function verifySignature(rawBody, signature, secret) {
  const expectedSignature = crypto
    .createHmac("sha256", secret)
    .update(rawBody)
    .digest("hex");

  return crypto.timingSafeEqual(
    Buffer.from(expectedSignature),
    Buffer.from(signature)
  );
}
```

This example shows the basic idea of comparing a calculated signature with the signature received in the request.

In production, you should also validate the timestamp and follow the exact signature format used by Whooks.

---

## Timestamp validation

Webhook signatures include a timestamp. The timestamp helps prevent replay attacks, where an attacker tries to resend an old webhook request.

Recommended approach:

- Extract the timestamp from `webhook-timestamp` header
- Compare it with the current server time
- Reject requests that are too old
- Use a short tolerance window (5 minutes is a good starting point)


---

## Replay protection

To reduce the risk of replay attacks:

- Validate the signature timestamp
- Reject old requests
- Store processed event IDs
- Use idempotency checks
- Monitor repeated delivery attempts

Signature verification confirms authenticity. Idempotency helps prevent repeated processing. Both are important.

---

## HTTPS

Always use HTTPS for webhook endpoints. HTTPS helps protect data in transit and prevents sensitive payloads from being exposed over insecure connections. Avoid using plain HTTP in production environments.

---

## Failed verification

If signature verification fails, reject the request. Recommended response:

```http
HTTP/1.1 400 Bad Request
```

or:

```http
HTTP/1.1 401 Unauthorized
```

Do not process the event if verification fails. You may log the failure for debugging and security monitoring, but avoid logging sensitive payload data.

---

## Best practices

- Always verify webhook signatures
- Always use the raw request body for verification
- Keep signing secrets private
- Rotate secrets periodically
- Validate timestamps when available
- Use HTTPS in production
- Reject invalid signatures
- Log verification failures carefully
- Combine signature verification with idempotency checks

---

## Common mistakes

Avoid these common issues:

- Processing webhooks without verifying signatures
- Using parsed JSON instead of the raw request body
- Exposing signing secrets in frontend code
- Logging secrets or sensitive payloads
- Ignoring timestamp validation
- Accepting old webhook requests
- Returning success for invalid signatures

---

## Summary

Webhook security ensures that your system only processes legitimate events from Whooks.

To secure your webhook endpoints:

- Verify the signature
- Use the raw request body
- Keep your signing secret secure
- Validate timestamps
- Use HTTPS
- Reject invalid requests

This protects your system from fake events, replay attacks, and unauthorized webhook traffic.
